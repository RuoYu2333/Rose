# Rose 引擎面试问答整理

> 说明：本文基于当前仓库 `RuoYu2333/Rose` 的代码与配置整理，尽量按“面试回答”的方式组织。部分问题仓库中尚未实现完整功能，因此答案会明确区分“已实现”与“理论上可这样回答”。

---

## 一、项目与定位

### 1. 先介绍一下 Rose 这个引擎的整体定位，它主要解决什么问题？

可以这样回答：

> Rose 目前更像是一个基础游戏引擎骨架，核心目标是把应用入口、窗口、输入、渲染、资源对象和调试界面这些基础设施搭起来，让上层能比较方便地进行游戏逻辑和渲染开发。它主要解决的是“把引擎底层能力标准化、模块化”的问题，包括窗口创建、事件分发、图形上下文初始化、2D 渲染提交、日志和断言等。

仓库里能看到的核心模块包括：
- `Application`
- `Layer`
- `Window`
- `Input`
- `Renderer` / `Renderer2D`
- `ImGuiLayer`
- `Log`

这说明它当前更偏向**引擎基础架构**，而不是完整商业引擎或完整编辑器。

### 2. 为什么你们要自研引擎，而不是直接用现成的引擎？

可以这样回答：

> 自研引擎的价值主要在于可控性和学习深度。现成引擎功能很完整，但底层架构和扩展方式是被定义好的；自研可以按照自己的需求去设计渲染流程、对象生命周期、Layer 架构以及调试方式，同时也能更深入理解图形 API、资源管理和引擎组织方式。对于这个项目来说，自研更适合作为学习和验证架构设计的载体。

### 3. Rose 目前主要偏 2D、3D，还是混合渲染？

可以这样回答：

> 目前 Rose 是偏 2D 的，但底层保留了通用渲染抽象，具备扩展到 3D 的基础。仓库里最明显的是 `Renderer2D`、`DrawQuad`、`DrawRotatedQuad` 这些接口，说明 2D 批处理是主要落点；同时也有 `Renderer::Submit`、`Shader`、`VertexArray`、`Camera` 等更通用的渲染抽象，可以支撑更底层的 3D 或自定义渲染。

### 4. 你在这个项目里主要负责哪部分？你做得最深的一块是什么？

仓库本身无法证明“你”个人负责的内容，所以面试时可以结合实际说：

> 如果从当前代码结构看，我理解最深的部分通常会是渲染层和基础架构层，比如 `Application` / `Layer` / `Renderer2D` / `ImGui` / 资源对象这些模块。因为这些模块直接决定引擎的组织方式、渲染路径和调试体验，也最能体现工程设计能力。

如果你确实参与了某一块，可以把它替换成实际负责内容。

### 5. 如果让你用三句话概括这个引擎的核心设计，你会怎么说？

可以这样答：

> 第一，Rose 用 `Application + Layer` 的方式组织主循环，把更新、事件和 UI 渲染统一到同一套框架里。第二，渲染层采用抽象接口和 2D 批处理思路，把上层逻辑和底层 OpenGL 细节隔离开。第三，资源与对象生命周期主要依赖 RAII、智能指针和集中式管理，配合日志、断言和 ImGui 方便调试。

## 二、架构设计

### 1. 你们的主循环是怎么设计的？逻辑帧和渲染帧如何拆分？

可以这样回答：

> Rose 目前采用单主循环的方式。`Application::Run()` 每一帧先计算 `TimeStep`，然后依次驱动各个 `Layer` 的 `OnUpdate`，再进入 ImGui 帧调用 `OnImGuiRender`，最后执行窗口更新和缓冲交换。当前没有做逻辑线程与渲染线程分离，也没有独立的逻辑帧/渲染帧调度器，逻辑和渲染还是在同一个线程里顺序执行。

代码上能看到：
- `Application::Run()` 里通过 `glfwGetTime()` 计算时间步长
- `for (Layer* layer : m_LayerStack)` 执行更新
- `m_ImGuiLayer->Begin()` / `End()` 管理 UI 帧
- `m_Window->OnUpdate()` 完成窗口刷新

### 2. 引擎中场景、实体、系统、资源管理器之间是什么关系？

可以这样回答：

> 当前仓库里还没有完整的 Scene / Entity / System / ResourceManager 体系。现阶段更多是用 `Layer` 作为功能组织单元，用 `Renderer` 和 `Renderer2D` 做渲染抽象，用 `Ref<T>` 管理资源对象。也就是说，Rose 目前还处于基础引擎架构阶段，尚未把 ECS 和完整场景管理系统做出来。

### 3. 你们怎么做模块解耦？模块之间通过什么方式通信？

可以这样回答：

> 主要通过接口抽象和事件系统来解耦。比如窗口、图形上下文和渲染 API 都做了抽象：`Window`、`GraphicsContext`、`RendererAPI`。模块间通信则依赖事件分发和 Layer 回调，比如 `OnEvent`、`OnUpdate`、`OnImGuiRender`。上层只关心这些接口，不直接依赖底层 OpenGL 或 GLFW 的细节。

### 4. 引擎有没有做渲染线程和逻辑线程分离？如果有，怎么同步数据？

可以这样回答：

> 目前没有。现在是主线程里顺序完成逻辑更新、渲染提交和 ImGui 绘制，没有看到独立的渲染线程或逻辑线程同步机制。如果后续要做线程分离，一般会把场景状态、渲染命令或者帧数据做双缓冲/命令队列，再通过同步原语保证提交时机。

### 5. 如果某个系统更新顺序出错，会造成什么问题？你怎么避免？

可以这样回答：

> 如果更新顺序出错，常见问题包括：相机状态没更新就开始渲染、输入状态过早或过晚被读取、UI 显示的数据和逻辑状态不一致、某些依赖前置系统的数据拿到旧值。避免方式是把主循环顺序固定下来，明确输入、更新、渲染、UI 的阶段边界，并通过 Layer 顺序、断言和日志来保证关键依赖关系不被破坏。

## 四、渲染管线

### 1. Rose 的 Render2D 负责什么？它更新哪些渲染态？

可以这样回答：

> `Renderer2D` 负责 2D 场景的批处理和提交工作。它维护批次相关的状态，例如当前顶点缓冲、索引计数、纹理槽、白纹理、渲染用的 `Shader`、`VertexArray` 等。上层只需要调用 `DrawQuad` 或 `DrawRotatedQuad`，内部会把这些图元组织成一批可提交给 GPU 的数据。

从 `Renderer2DData` 能看到：
- `QuadVertexArray`
- `QuadVertexBuffer`
- `TextureShader`
- `WhiteTexture`
- `QuadVertexBufferBase / Ptr`
- `TextureSlots`
- `QuadIndexCount`

### 2. 一个逻辑对象从数据变化到画面显示，中间经过哪些步骤？

可以这样回答：

> 先是 CPU 侧逻辑更新，比如相机、位置、旋转、颜色等状态更新。然后这些数据会进入渲染层，由 `Renderer` 或 `Renderer2D` 组装成顶点、索引、纹理槽和统一变量。最后渲染层把这些数据提交到 OpenGL，GPU 完成实际绘制。

### 3. 你们怎么做剔除、排序和批处理？

可以这样回答：

> 当前仓库里批处理已经有比较明确的实现，尤其是 `Renderer2D` 里对四边形的大批量顶点缓冲和纹理槽管理。但剔除和完整排序没有看到系统级实现，现阶段更偏向“先把渲染提交批起来”。如果后续做大场景，通常会继续补视锥剔除、透明排序和按材质/纹理排序。

### 4. 大量 Sprite 同屏时，如何减少 DrawCall？

可以这样回答：

> 主要思路是批处理。Rose 的 `Renderer2D` 通过共享的大顶点缓冲、统一的索引生成和纹理槽复用，把很多 Sprite 尽可能合成到同一批次里提交，从而减少 DrawCall 和状态切换。

### 5. 透明物体和不透明物体的渲染顺序怎么处理？

> 当前仓库没有专门的透明/不透明渲染队列实现。理论上，一般做法是先渲染不透明物体，再按深度从远到近渲染透明物体，以保证混合结果正确。

### 6. 你们的 GLSL shader 主要做了哪些事情？

可以这样回答：

> 当前 shader 主要做顶点变换、颜色/纹理传递和片元着色。顶点着色阶段会使用 `u_ViewProjection` 和 `u_Transform` 完成坐标变换，片元阶段根据颜色或纹理输出最终像素颜色。整体比较轻量，主要服务于 2D 和基础渲染示例。

### 7. 如果某个角色有翅膀扇动效果，你会放在逻辑层还是渲染层？为什么？

可以这样回答：

> 如果只是简单的周期性摆动，放在逻辑层更新 transform 就够了；如果涉及骨骼动画、顶点扰动或者材质层面的动画效果，更适合交给渲染层或者 shader 处理。核心原则是：谁最接近数据产生和使用，就放在哪一层。

## 五、内存与资源管理

### 1. Rose 里的内存管理策略是什么？

可以这样回答：

> Rose 目前的内存管理是按对象生命周期分层处理的。`Scope<T>` 对应独占所有权，`Ref<T>` 对应共享资源；窗口、OpenGL 资源、Layer 等对象主要通过 RAII 和统一生命周期管理释放；一些全局对象则以长期驻留方式存在。整体上优先保证所有权清晰和资源自动释放。

### 2. 为什么不用 shared_ptr 作为核心对象的默认方案？

可以这样回答：

> 因为 `shared_ptr` 会带来引用计数开销，也会模糊对象所有权。核心对象更适合明确由单一模块持有，比如 `Application` 持有 `Window`，这样销毁时机更明确，性能和架构边界也更清楚。`shared_ptr` 更适合确实需要共享的资源对象，比如 `Shader`、`Texture`、`VertexArray`。

### 3. 哪些对象适合对象池，哪些对象适合 unique_ptr？

可以这样回答：

> 对象池适合高频创建销毁、生命周期短、数量大的对象，比如粒子、子弹和临时特效；`unique_ptr` 适合被某个模块独占管理、生命周期清晰的对象，比如窗口、场景管理对象或者某些核心服务对象。当前仓库还没有对象池实现，但这就是理论上的划分方式。

### 4. 资源加载是怎么做的？同步还是异步？

可以这样回答：

> 当前仓库里的资源加载主要是同步的。比如纹理通过 `Texture2D::Create(path)` 直接加载，shader 通过 `Shader::Create(...)` 或 `ShaderLab::Load(...)` 立即创建并使用。这样实现简单、调试方便，但如果后面做大地图或大资源加载，可能需要进一步拆成异步 IO、解码和 GPU 创建几个阶段。

### 5. 大量临时对象如何避免频繁 new/delete？

可以这样回答：

> 当前仓库还没有内存池、arena allocator 或对象池等高级优化。现阶段主要依靠 RAII、栈对象、智能指针以及批处理过程中的集中分配来控制资源生命周期。后续如果临时对象很多，可以再考虑池化或分配器优化。

### 6. 资源释放和场景卸载怎么保证不泄漏？

可以这样回答：

> 核心原则是“谁申请谁释放”，并且尽量让资源跟随对象生命周期自动释放。像 OpenGL 纹理、buffer、shader 都在析构里做了对应的销毁；LayerStack 在析构时会统一 delete 层对象；资源对象则通过 `Ref<T>` 托管，引用归零后自然释放。这样可以尽量减少手动 `delete` 和泄漏风险。

## 七、性能与优化

### 1. 你们怎么定位引擎里的性能瓶颈？

可以这样回答：

> 我会结合 Visual Studio 的性能分析工具、日志和 ImGui 调试面板一起排查。VS 用来找 CPU/内存热点，日志记录资源加载和异常流程，ImGui 则用于实时观察帧率、状态和关键参数。这样能比较快地区分问题是在逻辑、渲染还是资源加载。

### 2. 如果渲染帧率下降 30%，你会先查哪里？

可以这样回答：

> 我会先判断是 CPU 端还是 GPU 端的问题，再看最近有没有改动渲染路径。CPU 侧重点查对象创建销毁、逻辑更新、渲染提交开销；GPU 侧重点查 DrawCall、shader 复杂度、纹理切换、过绘制和分辨率。若是资源同步加载导致卡顿，也要优先排查。

### 3. 如果同屏对象很多，除了合批你还会怎么优化？

可以这样回答：

> 除了合批，我还会做剔除、LOD、分帧更新、减少状态切换和优化数据布局。比如视锥剔除能直接跳过不可见对象，LOD 能降低远距离对象的成本，而更好的数据组织和对象池则能减少 CPU 的访问开销。

### 4. ECS 在性能上最大的收益是什么？

> 当前仓库没有 ECS 实现。理论上 ECS 最大的收益是更好的数据局部性和更适合批量处理，从而提高 CPU 缓存命中率，并让系统更新更容易并行化。

### 5. 你们有没有做过内存池、arena allocator 或其他优化？

> 没有看到这类实现。当前主要靠 RAII、智能指针和批处理的集中分配来管理资源。

### 6. 如果加载大地图卡顿，你会怎么拆分优化？

可以这样回答：

> 我会把大地图加载拆成多个阶段：先做资源读取和解析，再做实体/对象构建，最后做 GPU 资源创建；同时采用分块加载和按需加载，优先加载玩家附近区域，远处区域延迟加载。如果资源很大，还可以把磁盘 IO 和数据解码放到后台线程，把 GPU 创建保留在渲染线程。

## 八、工程与调试

### 1. 这个引擎有没有做工具链？比如资源导入、场景编辑、打包？

可以这样回答：

> 目前还没有完整工具链。仓库里有 `premake5.lua` 负责工程生成和编译配置，也有 `ImGuiLayer` 作为运行时调试 UI，但还没有看到资源导入器、场景编辑器、序列化/反序列化管线或打包系统。现在更像是引擎基础设施阶段。

### 2. 你们怎么做日志、断言和调试可视化？

可以这样回答：

> 日志使用 `spdlog`，区分 core logger 和 client logger；断言通过 `RS_ASSERT` 和 `RS_CORE_ASSERT` 完成，失败时会输出信息并触发 `__debugbreak()`；调试可视化则通过 ImGui 实现，`ImGuiLayer` 在每帧创建和渲染 UI，Layer 也可以在 `OnImGuiRender()` 里输出自己的调试面板。

### 3. 如果某个系统输出结果错误，你怎么快速定位？

可以这样回答：

> 我一般会先拆成输入、处理中间状态和最终输出三段来排查。先用日志和断言确认数据在哪一步开始异常，再用 ImGui 把关键中间值实时展示出来，必要时通过断点和单层隔离的方式逐步缩小范围。因为 Rose 的 Layer 结构比较清晰，所以把问题缩到某个 Layer 或某个渲染路径会比较直接。

### 4. 你们有没有做过状态快照、回放或者调试重放？

> 当前仓库没有看到这类功能。

### 5. 如果引擎要支持新平台，你觉得哪一层最先要改？

可以这样回答：

> 我会先改平台抽象层，而不是上层业务逻辑。当前代码里最需要适配的是 `Window`、`Input`、`GraphicsContext` 和 `RendererAPI` 这些平台相关层；上层的 `Application`、`Layer`、`Renderer` 和 UI 逻辑尽量保持不变，这样迁移成本最低。

## 补充：当前仓库中比较关键的实现点

### 1. Application + Layer 框架
- `Application` 负责主循环、事件、窗口和 UI 驱动
- `Layer` 负责模块化功能组织
- `LayerStack` 统一管理层级并负责销毁

### 2. 渲染抽象
- `Renderer` 是更底层的通用提交接口
- `Renderer2D` 是 2D 批处理层
- `RenderCommand` 层统一封装渲染 API 选择

### 3. 资源与对象生命周期
- `Scope<T>` = `unique_ptr<T>`
- `Ref<T>` = `shared_ptr<T>`
- OpenGL 资源在析构时销毁
- `LayerStack` 析构统一 delete Layer

### 4. 调试与工程配置
- `spdlog` 负责日志
- 断言宏用于调试期快速停止
- `ImGuiLayer` 负责运行时 UI
- `premake5.lua` 负责工程生成与配置

## 最后可直接背诵的总述

> Rose 目前是一个基于 C++ 和 OpenGL 的基础游戏引擎骨架，核心已经包含应用框架、Layer 分层、窗口与输入、日志断言、ImGui 调试层，以及 2D 批渲染和基础渲染抽象。它的设计重点是把引擎底层能力模块化、把资源生命周期理清楚，并为后续扩展 ECS、编辑器、资源管线和更多平台做基础。当前仓库还没有完整的 ECS、Lua 运行时、工具链和多线程渲染，但作为基础引擎结构已经比较清楚。
